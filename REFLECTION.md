## System Explanation

### Architecture and agent roles

The system uses the **Orchestrator Pattern** with five agents:

| Agent | Role |
|-------|------|
| **Orchestrator** | Single entry point; delegates to specialists; composes the final customer reply |
| **Request Parser / Item Mapper** | Maps colloquial product names to exact catalog `item_name` values |
| **Inventory Agent** | Checks stock as of `request_date`; flags shortfalls and restock needs |
| **Quoting Agent** | Prices line items using quote history and bulk discounts |
| **Ordering Agent** | Validates cash, records `stock_orders` and `sales` transactions |

All inter-agent communication is plain text. The Orchestrator is the only component that invokes specialists; workers never coordinate peer-to-peer. This keeps a single control flow, prevents conflicting database writes, and produces one coherent customer response per request.

See the [high-level architecture diagram](./WORKFLOW.md#high-level-architecture) and [per-request sequence](./WORKFLOW.md#per-request-sequence) in WORKFLOW.md.

### Key design decisions

1. **Dedicated Parser agent** — Sample requests use colloquial names ("8.5x11 colored paper", "poster boards") that must map to exact catalog entries before any database operation. A separate parser keeps mapping logic isolated from inventory and pricing.

2. **Parallel Inventory + Quoting** — Once items are parsed, stock checks and quote generation run concurrently via `ThreadPoolExecutor`. These steps are independent and both need the parsed line items, so parallel execution reduces latency without changing correctness.

3. **Single-writer Ordering agent** — Only the Ordering Agent calls `create_transaction()`. This avoids write conflicts and makes cash validation a single checkpoint before restock or sale.

4. **Date-scoped queries** — Every handoff includes `request_date`. Inventory, cash, and transaction tools scope queries to that date so sequential test scenarios in `quote_requests_sample.csv` reflect realistic state evolution.

5. **Restock-before-fulfill** — When stock is insufficient and cash allows, the Ordering Agent places `stock_orders` before recording `sales`, with supplier delivery ETA from `get_supplier_delivery_date()`.

### Per-request pipeline

1. Customer request + `request_date` → Orchestrator
2. Orchestrator → Parser: parse and map items
3. If UNRESOLVED items → clarification reply to customer
4. Orchestrator → Inventory + Quoting (parallel): availability report + priced quote
5. If restock needed and cash insufficient → failure report
6. Orchestrator → Ordering: fulfill + optional restock
7. Orchestrator → Customer: unified quote + order confirmation

Routing branches (clarification, restock, failure) are documented in [WORKFLOW.md routing decisions](./WORKFLOW.md#routing-decisions).

## Evaluation Results

The system was evaluated against the full `quote_requests_sample.csv` dataset (20 requests). Results are in [test_results.csv](./test_results.csv), generated by `python project_starter.py`.

### Summary metrics

| Metric | Result |
|--------|--------|
| Requests processed | 20 / 20 |
| Successful fulfillments (`ORDER_STATUS: success`) | 12 |
| Partial fulfillments | 1 (request 13) |
| Clarification / unresolved | 6 |
| Other (non-terminal status) | 1 (request 14, `ORDER_STATUS: pending`) |
| Requests that changed cash balance | 12 |
| Cash after first fulfilled request | $45,134.70 |
| Final cash balance | $44,876.70 |
| Final inventory value | $4,236.55 |

### Outcome breakdown

- **12 success** — Full order processed with sales transactions recorded (requests 1, 4, 5, 6, 7, 8, 10, 11, 12, 17, 18, 19).
- **1 partial** — Request 13 (gathering order): insufficient cardstock stock; system placed restock orders, recorded partial sales, and returned `ORDER_STATUS: partial` with delivery ETA.
- **6 clarification** — Parser could not map items to catalog (requests 2, 3, 9, 15, 16, 20). Examples: "balloons", "cardboard for signage", "posters", "tickets". Orchestrator returned a clear request for clarification instead of guessing.
- **1 pending** — Request 14 (performance order): Ordering Agent recorded restock transactions but returned `ORDER_STATUS: pending` rather than success or partial; cash balance still changed due to supplier orders.

### Strengths

1. **Realistic fulfillment mix** — Not all requests succeed. The system refuses or clarifies when items are unknown or stock/cash constraints apply, which matches real business behavior.

2. **Competitive quoting with rationale** — The Quoting Agent applies bulk discounts and documents reasoning. Request 4 includes a `QUOTE_RATIONALE` with a 10% bulk discount on a reception order; request 19 applies a 10% exhibition discount informed by quote history.

3. **Restock path works** — When inventory is short but cash is sufficient, the Ordering Agent places supplier restocks before sales and reports delivery ETAs. Request 19 (large exhibition order) restocked three items and fulfilled all line items successfully.

4. **Financial reporting tool in use** — The Ordering Agent invoked `get_financial_report` during order processing (17 times in the execution log), confirming the post-remediation tool wrapper is active.

5. **No internal error leakage** — Customer responses do not expose `SPECIALIST_ERROR`, API failures, profit margins, or other sensitive internal details.

6. **Date-aware state evolution** — Processing requests sorted by date produces meaningful cash and inventory changes across the test run (12 requests affecting cash balance).

## Suggestions for Further Improvement

### 1. Reduce UNRESOLVED rate with synonym mapping

Six of 20 requests ended in clarification because the Parser could not map colloquial terms to catalog names. Many are close matches the LLM could resolve with explicit rules:

- "heavy cardstock" → `Cardstock` or `250 gsm cardstock`
- "poster boards" → `Large poster paper (24x36 inches)`
- "cardboard for signage" → `Cardstock` or `100 lb cover stock`

Adding synonym hints to Parser instructions or a lightweight fuzzy-match pre-step would convert more inquiries into quotable orders without inventing catalog items.

### 2. Explicit partial-order customer messaging

Request 13 was fulfilled partially (restock + partial sale) but the customer reply still opens with "Thank you for your order" without clearly stating what was and was not fulfilled. The Orchestrator should detect `ORDER_STATUS: partial` and compose a dedicated message explaining available vs. backordered quantities and expected delivery dates.

### 3. Sanitize customer-facing output format

Successful replies currently include raw agent structured fields (`TRANSACTIONS`, `txn_id`, `ORDER_STATUS`). A final formatting step in the Orchestrator could translate these into plain-language confirmations while keeping internal details out of the customer view.
